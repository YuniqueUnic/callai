### 1. 配置备份与恢复流程

callai 的配置备份策略遵循**「自动 + 手动 + 安全」**原则，防止用户误操作或文件损坏导致数据丢失。

#### 备份策略

**自动备份机制**（推荐默认开启）：

- **启动时自动备份**：每次应用启动时，检测 `config.toml` 是否存在且可解析。
  - 如果解析成功 → 在 `backups/` 目录下生成带时间戳的备份（最多保留最近 10 个）
  - 如果解析失败 → 自动使用最近一次成功的备份，并提示用户

- **保存时自动备份**：用户每次保存配置前，先将当前配置备份一次。

**备份文件命名规则**：
```
config.toml.2026-07-11_14-30-00.bak
config.toml.2026-07-10_09-15-00.bak
```

**备份位置**：
```
~/.config/callai/backups/
```

#### 恢复流程

**场景 A：启动时检测到配置损坏（自动恢复）**

1. 应用启动时尝试解析 `config.toml`
2. 解析失败 → 弹出温和提示：
   > 「配置文件好像遇到了点小问题… 我们已经自动使用了上次的备份版本～」
3. 自动加载最近一次成功的备份
4. 将损坏的原文件重命名为 `config.toml.corrupted` 保留
5. 提供按钮：「查看损坏文件」 / 「继续使用备份」

**场景 B：用户手动恢复备份**

1. 进入 **设置 → 高级 → 配置管理**
2. 显示最近 10 个备份列表（带时间和文件大小）
3. 用户选择某个备份 → 点击「恢复此版本」
4. 弹出确认：
   > 「确定要恢复到这个版本吗？当前配置会被覆盖哦～」
5. 恢复成功后提示并重启应用（或热重载）

**场景 C：完全重置配置**

- 设置中提供「重置为默认配置」按钮
- 执行后会自动备份当前配置，再生成全新默认配置

#### 备份相关设置（设置页中）

- [ ] 启动时自动备份配置（默认开启）
- 保留备份数量：`10` 个（可修改）
- 手动备份按钮（立即备份当前配置）

### 2. 多语言（i18n）文案管理方案

callai 计划支持**中英双语**起步，后续可扩展。采用**前后端分离管理**的方案。

#### 整体架构

| 部分       | 推荐方案                    | 文件格式     | 说明 |
|------------|-----------------------------|--------------|------|
| **前端**   | `i18next` + `react-i18next` | JSON         | 主要文案、UI 提示、Toast、弹窗 |
| **后端**   | `rust-i18n` 或 `fluent`     | `.ftl` / JSON | 错误信息、日志中的文案 |
| **共享**   | 通过 Tauri Command 传递     | -            | 后端错误消息翻译后返回给前端 |

#### 前端 i18n 方案（推荐）

**技术栈**：
- `i18next`
- `react-i18next`
- `i18next-browser-languagedetector`（自动检测语言）

**文件结构**：

```
src/i18n/
├── locales/
│   ├── en/
│   │   ├── common.json
│   │   ├── alarms.json
│   │   ├── logs.json
│   │   └── settings.json
│   └── zh-CN/
│       ├── common.json
│       ├── alarms.json
│       ├── logs.json
│       └── settings.json
├── i18n.ts
└── index.ts
```

**示例 `alarms.json`**（部分）：

```json
{
  "create_alarm": "新建闹钟",
  "alarm_name": "闹钟名称",
  "next_trigger": "下次触发",
  "save_success": "闹钟设置好啦～以后会按时帮你照顾 AI 的！",
  "binary_not_found": "这个命令好像不在本机哦～要检查一下路径吗？",
  "execution_failed": "这次任务没顺利完成，我们已经准备好再试一次啦！"
}
```

**语言切换**：
- 设置页提供语言切换器（下拉或 Segmented Control）
- 切换后立即生效（无需重启）
- 使用 `localStorage` 记住用户选择

#### 后端（Rust）多语言方案

推荐使用 **`rust-i18n`**（轻量且易用）：

**优点**：
- 支持 `.ftl`（Fluent）或 JSON
- 与前端 i18n 风格接近
- 可以方便地根据 `Accept-Language` 或用户设置返回对应语言的错误信息

**实现方式**：

1. 后端定义错误枚举
2. 通过 Tauri Command 返回时，附带当前语言
3. 前端根据语言展示对应文案（推荐做法）

或者更简单的方式：
- 后端只返回**错误码**（如 `ERR_BINARY_NOT_FOUND`）
- 前端根据错误码 + 当前语言展示可爱文案

**推荐做法**：**错误码 + 前端文案映射**（更灵活，也更符合可爱风格）

#### 文案管理建议

- 所有用户可见文案统一放在前端 JSON 中管理
- 后端仅返回结构化数据 + 错误码
- 新增语言时，只需在 `locales/` 下新增对应文件夹即可
- 建议使用 **i18next 的命名空间**（按页面/模块拆分），方便后期维护

**示例错误码映射**（前端）：

```ts
const errorMessages = {
  ERR_BINARY_NOT_FOUND: t('alarms.binary_not_found'),
  ERR_EXECUTION_FAILED: t('alarms.execution_failed'),
  // ...
}
```

#### 语言切换流程

1. 用户在设置页切换语言
2. 调用 `i18next.changeLanguage(lang)`
3. 保存到 `localStorage`
4. 应用内所有文案实时更新
5. 可选：把语言偏好通过 Tauri 传给后端（用于日志等）

### 总结

| 模块             | 方案                          | 优点 |
|------------------|-------------------------------|------|
| **配置备份**     | TOML + 时间戳备份 + SQLite    | 安全、可恢复、用户可手动编辑 |
| **恢复流程**     | 自动 + 手动双模式             | 用户友好，降低使用风险 |
| **i18n**         | 前端 `i18next` + 错误码机制   | 灵活、易维护、前后端解耦 |
| **文案风格**     | 温和 + 可爱语气词             | 符合动森品牌调性 |
